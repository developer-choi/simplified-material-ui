# Material UI
> Material-UI의 방대한 코드에서 핵심 로직만을 추출하여 재구성한 분석 프로젝트입니다.

## 목적
Material-UI(MUI)는 IE11 등 레거시 브라우저 지원과 수많은 엣지 케이스 대응을 위해 코드의 복잡도가 매우 높습니다.

이 프로젝트는 프로덕션 레벨의 검증된 코드를 분석한 뒤, **사내 개발 환경에 맞춰 최소화**하여 추후 활용하는 것을 목표로 합니다.

## 배경
기능을 구현할 때 단순히 뇌피셜에 의존하여 만들 수도 있습니다.

하지만 저는 항상 **'최고점'의 결과물**을 만들어내려는 개발 습관을 지향합니다.

검증된 오픈소스 분석을 통해 시행착오를 줄이고 설계의 완성도를 높입니다.

## 활용 방법

### 1. 인터페이스 설계의 기준점
디자이너가 요구한 스펙 외에도, **미래에 요구될 수 있는 기능**을 예측하는 데 MUI의 Props를 참고합니다.

MUI에 존재하는 Props는 수많은 프로젝트에서 검증된 **'보편적인 요구사항'**의 집합체이기 때문입니다.

* **예시:** 버튼의 배경색을 바꿀 수 있는 `color` 속성을 요구받은 경우
  * MUI를 참고하여 `variant` (contained/outlined/text) 속성의 존재를 확인합니다.
  * 디자이너에게 "추후에 테두리만 있는 스타일이나 텍스트만 있는 스타일도 필요할까요?"라고 선제적으로 질문합니다.
  * 이를 통해 향후 변경 사항에 유연하게 대처할 수 있는 인터페이스를 미리 설계합니다.

### 2. 놓치기 쉬운 UX 디테일 확보
기능 구현을 넘어, 사용자 경험의 디테일을 챙기는 참고 자료로 활용합니다. "있으면 분명히 좋은 기능"은 원본 코드를 분석하지 않고는 놓치기 쉽습니다.

* **예시:** 모달이 닫힐 때, 모달을 열었던 버튼으로 **포커스를 복원** 해주는 로직

### 3. 컴포넌트 확장성 확보
MUI를 벤치마킹하여, 중복코드를 줄이고 확장 가능한 컴포넌트 시스템을 구축합니다.

* **구조적 재사용 예시:**
  * `Modal`, `FocusTrap`, `Portal`과 같은 저수준(Low-level) 컴포넌트를 먼저 구현합니다.
  * 이를 조합하여 `Dialog`, `Drawer` 등 고수준 컴포넌트에서 재활용합니다.
  * **결과:** 모든 오버레이 컴포넌트에서 딤(Dim) 처리, 스크롤 잠금, 포커스 트랩 등의 동작을 일관되게 유지할 수 있습니다.

## 분석 및 단순화 기준
원본 코드를 분석할 때 다음 기준에 따라 코드를 분류하고 처리합니다.

1.  **삭제:**
  * IE11 등 레거시 브라우저 지원 코드
  * 현대적인 React 패턴으로 대체 가능한 레거시 라이프사이클
  * 내 프로젝트 스펙에 불필요한 과도한 추상화
2.  **유지:**
  * 기능의 핵심 로직
  * 의도가 명확하지 않아 삭제 시 사이드 이펙트가 우려되는 코드 (보수적 접근)
