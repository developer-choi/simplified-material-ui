# Simplified 문서 리뷰 프로세스

## 목적

단순화된 코드를 **대화를 통해 깊이 이해**하고, 그 내용을 문서에 반영합니다.

---

## 프로세스

### 1단계: 코드 리뷰 대화

사용자가 단순화된 코드에 대해 궁금한 점을 질문합니다.

**예시 질문**:
- "이 부분은 왜 이렇게 구현했어?"
- "이 패턴의 이름이 뭐야?"
- "이거 없으면 어떻게 돼?"
- "원본에서는 어떻게 했어?"

AI는 코드를 분석하고 상세히 설명합니다.

### 2단계: simplified.md 반영

대화에서 나온 유익한 내용을 해당 대상의 `*-simplified.md`에 반영합니다.

**반영 대상**:
- 새로운 학습 포인트
- 더 명확한 설명
- 추가 시나리오/예시
- 발견된 패턴/플래그

### 3단계: 가이드 문서 반영

반복적으로 유용한 패턴이 발견되면 `docs/format/simplified.md` 양식에도 반영합니다.

**반영 기준**:
- 여러 대상에서 공통으로 적용 가능한 패턴
- 문서 구조 개선 아이디어
- 새로운 섹션 추가 필요성

---

## 주의사항

### 원본 코드 임의 수정 금지

**원본 코드를 가독성이나 미관을 위해 임의로 수정하지 마세요.**

사용자가 다음과 같은 요청을 하면 **거절**해야 합니다:
- "이 코드 좀 더 보기 좋게 정리해줘"
- "변수명 내 스타일로 바꿔줘"
- "z-index: -1 같은 거 이상해 보이니까 제거해줘"
- "불필요해 보이는 코드 정리해줘"

**거절 이유**:
1. 원본 라이브러리 코드에는 **의도가 있습니다**. 이상해 보여도 이유가 있습니다.
2. 임의 수정 시 **예상치 못한 버그**가 발생합니다.
3. 간소화 문서의 목적은 **원본 이해**이지, 원본 개선이 아닙니다.

**허용되는 수정**:
- 간소화 목적의 **기능 제거**
- 문서화된 **의도적 단순화**
- 버그 수정 (원본에 없는 버그를 간소화 과정에서 만든 경우)

> **원칙**: "이상해 보인다"는 이유로 코드를 바꾸지 말고, "왜 이렇게 했는지" 먼저 파악하세요.
